# セキュリティ指針 - multi-agent-ojousama

本プロジェクトを安全に運用するための指針をまとめる。各エージェント（お嬢様 / 家政婦 / メイド）は本書に従うこと。

---

## 1. queue/ と workspace/ の取り扱い

`queue/` と `workspace/` は、エージェント間通信および作業領域として使われる。
これらに **機密情報を書き込んではならない**。

### 禁止事項

- API キー、トークン、パスワード、秘密鍵を YAML / 作業ファイルに直接書く
- あなたの個人情報、メールアドレス、住所等を記録する
- 顧客情報・本番データ等を作業領域にコピーする

### 許可事項

- タスク指示・状態・サマリの記述
- スクリーニング済みの公開情報
- ダミーデータ・サンプルデータ

`queue/*.yaml` および `workspace/*` は `.gitignore` でリポジトリから除外されているが、
ローカルディスク上には残るため、機密情報の混入は二次流出のリスクとなる。

---

## 2. シェル注入対策

`scripts/` 配下のシェルスクリプトを書く・改修する際の必須ルール。

### 必ず守ること

- **`eval` 禁止**: 文字列を実行コードに変換しない。
- **変数は必ず引用符で囲む**: `"$var"` のように二重引用符で囲み、空白・特殊文字による分割を防ぐ。
  - 悪い例: `tmux send-keys -t $target "$msg"`
  - 良い例: `tmux send-keys -t "$target" "$msg"`
- **ユーザー入力をそのままコマンドに渡さない**: `set -u` を有効化し、未定義変数を検出する。
- **配列展開は `"${arr[@]}"` を使う**: `${arr[*]}` や引用符なしは禁止。
- **`set -euo pipefail`** を冒頭に置く（早期失敗・パイプ失敗の捕捉）。

### tmux send-keys の注意

メッセージにシェルメタ文字（`` ` ``, `$`, `;`, `|` など）が含まれると、
受信側の Claude Code に意図しない解釈をされる可能性がある。送信前にサニタイズすること。

## 3. シークレット管理

### 配置場所

- API キー・トークン類は **`config/ntfy_auth.env`** などのドット env 形式で配置する。
- `config/` 配下の env ファイルは `.gitignore` で除外する。
- 環境変数経由の注入を推奨（`export ANTHROPIC_API_KEY=...`）。

### 禁止事項

- リポジトリにコミットされる `instructions/`, `scripts/`, `CLAUDE.md`, `README.md` などにキーを書く
- `queue/`, `workspace/`, `logs/` にキーを書く（ローカル残存の問題）
- コミットメッセージや PR 説明にキーを貼る

### コミット前チェック

- `gitleaks` を導入済み（`.gitleaks.toml` 参照）。コミット前に走らせるとよい。
- 既に push してしまった場合は、即座にトークン無効化 → ローテーション。履歴削除のみでは不十分。

## 4. ntfy_topic の秘匿性

ntfy（外部通知サービス）を利用する場合、`topic` 名は **推測困難な文字列** を用いること。

### 推奨

- 32 文字以上のランダム英数字（例: `ojousama-7f3k2m9p4q8r5s1t6v...`）
- 用途別にトピックを分離（あなた宛通知用・デバッグ用 など）

### 禁止

- `ojousama`, `test`, `notify` などの推測可能な単一語
- ユーザー名や日付をそのまま使う
- 公開チャンネル（README やコミットメッセージ）でトピック名を晒す

ntfy のデフォルトトピックは認証なしで誰でも購読可能。トピック名の秘匿性が
そのままアクセス制御の強度になる。漏洩した場合は別トピックに切り替えること。

---

## 5. インシデント発生時

- API キー漏洩を発見したら、まず該当キーを無効化し新キーを発行する。
- 漏洩経路（コミット / ログ / queue / workspace）を特定し、痕跡を除去する。
- 必要に応じてあなたに報告（家政婦経由）。

---

## 6. 報告チャネル

セキュリティ上の問題を発見した場合は、queue/ 経由ではなく、
あなたに直接（または信頼できる経路で）報告すること。
queue/ 内に書くと他エージェントから読まれる可能性がある。
